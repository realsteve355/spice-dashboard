## 1. Governance.sol — Multi-Candidate Redesign

### Motivation
The old Governance.sol allowed only one candidate per election per role. Any attempt to
nominate a second candidate reverted with "Gov: election already active for this role".
The system was unusable for real contested elections.

### Old model (single-candidate)
```
nominateForElection(role, candidate) → opens election with one candidate
vote(electionId, bool support)        → binary for/against
finaliseElection                      → pass if votesFor > votesAgainst
executeElection                       → after 7-day timelock
```

### New model (multi-candidate)
```
openElection(role) → opens nomination window, returns electionId
nominateCandidate(electionId, candidate) → adds a candidate (any citizen, multiple allowed)
vote(electionId, candidate)              → vote for specific candidate (plurality)
finaliseElection(electionId)             → determines winner by highest votes; cancels if no candidates
executeElection(electionId)              → after timelock, installs winner
resign(role)                             → incumbent vacates immediately (no election required)
```

### Timing constants (testnet values — reduced for testing)
```solidity
uint256 public constant NOMINATION_WINDOW =  5 minutes;
uint256 public constant VOTING_WINDOW     = 15 minutes;
uint256 public constant TIMELOCK          =  5 minutes;
uint256 public constant TERM              = 365 days;
uint256 public constant OBLIGATION_EXPIRY = 30 days;
```
> On mainnet these should be 7 days / 14 days / 7 days respectively.

### New Election struct
```solidity
struct Election {
    uint8   role;
    address openedBy;
    uint256 openedAt;
    uint256 nominationEndsAt;
    uint256 votingEndsAt;
    uint256 timelockEndsAt;    // 0 until finalised with a winner
    address winner;            // zero address until finalised
    bool    executed;
    bool    cancelled;         // set true on finalise with no candidates or draw
}
```

New mappings: `electionCandidates[id][]`, `candidateVotes[id][addr]`, `isCandidate[id][addr]`  
New views: `getCandidates(uint256) → address[]`, `getCandidateVotes(uint256, address) → uint256`

## 3. Votes.jsx — Complete Rewrite

Rewritten from scratch for the multi-candidate model.

**Key design decisions:**

**Two-provider approach (stale RPC fix):**
- `load()` uses public RPC (`base-sepolia-rpc.publicnode.com`) — for initial page load only
- `loadElections()` uses MetaMask signer provider — called after every write tx (always on latest block)
- Background `setTimeout(() => load(), 4000)` refreshes from public RPC after writes

**Receipt-based optimistic injection:**
After `openElection()` tx confirms, parse `ElectionOpened` event from receipt and inject
the new election into state immediately — before the public RPC has indexed the block.

**Ghost election guard:**
After any RPC read of `gov.elections(id)`, check `e.openedBy !== ethers.ZeroAddress`
before using the data. Empty structs return from unharvested RPC reads and must be discarded.

**Election status machine (client-side):**
```js
function electionStatus(e, nowSec) {
  if (e.executed)                                         return 'EXECUTED'
  if (e.cancelled)                                        return 'FAILED'
  if (e.timelockEndsAt > 0 && nowSec >= e.timelockEndsAt) return 'EXECUTE_READY'
  if (e.timelockEndsAt > 0)                               return 'TIMELOCK'
  if (nowSec > e.votingEndsAt)                            return 'FINALISE_READY'
  if (nowSec > e.nominationEndsAt)                        return 'VOTING'
  return 'NOMINATING'
}
```
Status recalculated every 15s via `setInterval` so phase transitions appear without refresh.

**Gas override:** All write txs use `const GAS = { gasLimit: 300000 }` — required on Base Sepolia
where gas estimation consistently fails with "missing revert data" for some contract calls.

**UI flow:** MCC Board panel (role holders + Resign button) → election list per role →
nomination form with citizen dropdown (when citizens loaded) + address text input fallback.

## 4. Citizen Name Lookup — Server-Side API

### Problem
Browser-side `getLogs` for `CitizenJoined` events consistently failed across multiple RPC
strategies (publicnode.com and sepolia.base.org) with LavaMoat errors and timeout issues.

### Solution: GToken contract reads via Vercel serverless function
**`colony-app/api/citizens.js`** — new GET endpoint:
1. Calls `colony.gToken()` to get GToken address
2. Calls `gToken.nextTokenId()` to know how many tokens have been minted
3. Loops `gToken.ownerOf(tokenId)` + `gToken.citizenName(tokenId)` for each tokenId
4. Returns `{ citizens: [{ address, name }] }` sorted by name

**`colony-app/src/utils/fetchCitizens.js`** — shared client utility:
```js
export async function fetchCitizens(colonyAddr) {
  const res = await fetch(`/api/citizens?colony=${colonyAddr}`)
  const { citizens } = await res.json()
  return citizens || []
}
```

Used by: Votes.jsx, Assets.jsx, RegisterCompany.jsx, Company.jsx (imported as `fetchColonyCitizens`).

GToken has `nextTokenId` (starts at 1, increments on each mint), `ownerOf(tokenId)`, and
`citizenName(tokenId)` — all public. No ERC-721Enumerable needed. Works regardless of
event log availability.
